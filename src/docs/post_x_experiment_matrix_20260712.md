# post_x 実験結果まとめ

## 目的

X への投稿について、以下のパターンが通るかを確認した。

- 通常投稿
- URL 付きの親投稿
- 親投稿への reply
- 既存ツイートへの reply
- URL あり・なしの違い
- 近い文面の再投稿

## 確認できたこと

| パターン | 結果 | 判定 | 補足 |
|---|---:|---|---|
| 通常投稿のみ | 成功 | 可能 | `reply` なしなら通る |
| URL付きの親投稿 | 成功 | 可能 | URL は `t.co` に変換された |
| 自分で作った親投稿への reply | 成功 | 可能 | reply 機能自体は通る |
| 既存の親ツイートへの reply | 失敗しがち | 要注意 | `403 Forbidden` が出た |
| URLなしの reply | 成功したことあり | 可能 | ただし文面が近いと止まることがある |
| ほぼ同文の再投稿 | 失敗 | 不可 | `duplicate content` で弾かれた |
| AI生成の reply をそのまま送る | 条件次第 | 可能 | 文字数や重複を避ければ通る余地あり |
| OAuth 1.0a の通常投稿 | 成功 | 可能 | 認証そのものは生きている |
| OAuth 1.0a の reply 投稿 | 成功/失敗が分かれる | 条件付き | 親ツイートと文面の条件が効いている |

## 実験の要点

- `reply` 機能そのものは使える
- ただし、**既存の親ツイート** に reply すると弾かれることがある
- **自分で新規投稿した親ツイート** に reply する形は通った
- 親投稿に URL を付ける運用は可能
- reply 側は URL を付けない方が扱いやすい
- 直近と近すぎる文面は `duplicate content` で弾かれることがある

## 現時点の結論

今の運用で現実的なのは次の形。

1. 親投稿を新規で立てる
2. 親投稿には URL を付ける
3. その親投稿に reply する
4. reply 側は URL を付けない
5. reply 文面は毎回少し変える
