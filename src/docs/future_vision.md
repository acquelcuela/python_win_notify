## 方向性

いまの Windows バッチは、将来的に AWS のサーバーレス構成へ移すことで、ローカル PC への依存を減らし、運用コストも抑えられます。

想定する形は次のとおりです。

- `EventBridge Scheduler` が `07:00`、`09:30`、`12:15`、`22:45` JST に起動する
- `Lambda` が Python のバッチ処理を実行する
- `S3` に HTML や JSON の生成物を保存する
- `SES` でメールを送る
- `Secrets Manager` か `SSM Parameter Store` に API キーやメール関連の秘密情報を置く

## 追加候補の優先順位

今後の追加候補は、`post_x_note` を先に進めて、`stock_x_trends` は後回しにするのが自然です。

- `post_x_note` は、note 記事の取得、除外、直近投稿済み回避という流れがはっきりしている
- `stock_x_trends` は、X の検索品質や Grok の出力品質に依存する度合いが大きい
- まずは「投稿の自動化」を固めた方が、運用上の価値が見えやすい

## この形が合う理由

- このバッチは、開始と終了がはっきりした短い定期処理だからです
- 外部データを集めてレポートを作り、そのまま送る流れは、状態を持たないサーバーレス処理と相性がよいです
- 実行回数が少ないので、AWS 側の実行コストも小さく収まりやすいです

## 期待できる利点

- 常時起動のローカル PC が不要になる
- スケジュールの安定性が上がる
- 遠隔からの運用がしやすくなる
- 基盤まわりの管理負担が減る
- データ収集、レポート生成、送信の役割分担が分かりやすくなる

## 技術メモ

- このプロジェクトは `yfinance` などの依存があるので、zip 配布よりコンテナイメージの方が扱いやすい可能性がある
- 最初は VPC に入れず、NAT コストを避ける方がよい
- 既存の `main.py` の流れは、1つの Lambda エントリポイントにしやすい
- 秘密情報はコードやコンテナに埋め込まず、Lambda の環境変数か `Secrets Manager` に分けて置く
- `GEMINI_API_KEY` やメールの秘密情報は `Secrets Manager` 優先、`MODEL_NAME` などの公開設定は環境変数で十分

## post_x_note の位置づけ

`post_x_note` は、note 記事を X に投稿する自動化モジュールの候補です。  
`tweet_tool` にある既存実装がかなり近く、特に次の部品はそのまま流用しやすいです。

- note の全記事取得
- ローカルキャッシュ
- 除外キー
- 直近投稿済みの重複回避

このため、`post_x_note` は最初に着手しやすいです。

### `post_x_note` を先にやる理由

- 既存コードの流用余地が大きい
- 仕様が比較的まっすぐで、結果が見えやすい
- note 側の取得と除外が動けば、運用の自動化効果がすぐ出る
- X 投稿だけに責務を絞れるので、切り分けが明快

### `post_x_note` の現状に合わせた考え方

- note 記事の最新取得は RSS か note API のどちらかに寄せる
- 投稿済み判定は「記事URL」と「投稿日時」を持っておけば十分
- 除外はファイル管理か SQLite 管理にまとめる
- 1日2回などの上限管理は、投稿履歴から判定できる
- `post_x_note` の投稿時刻は `07:15` と `21:00` の2回を想定する。朝は本処理の `07:00` と少しずらし、夜は1回だけにする。

## stock_x_trends の位置づけ

`stock_x_trends` は、X 上の話題銘柄やキーワードを拾う発見型のモジュール候補です。  
`post_x_note` と違って、外部検索品質のばらつきが大きいので、後回しが妥当です。

### `stock_x_trends` を後回しにする理由

- X 検索や Grok の返答品質に依存する
- 取得結果の再現性が低くなりやすい
- トークンや検索回数のコスト見積もりが先に必要
- まずは投稿系の基盤を固めた方が、全体の運用設計が安定する

### `stock_x_trends` の現状に合わせた考え方

- 既存の注目銘柄や市場概況とは別系統の「発見」機能として扱う
- ウォッチリストの延長ではなく、情報源の追加として設計する
- 実装するなら、まずは出力を JSON に限定して後段で整形する

## 移行の順番案

1. `post_x_note` を先に固める
2. レポート生成を Lambda から呼べる形にする
3. 生成物の保存先をローカル `src\output` から `S3` に変える
4. Gmail SMTP を `SES` に置き換える
5. 秘密情報を AWS の管理領域に移す
6. `stock_x_trends` は最後に回す
