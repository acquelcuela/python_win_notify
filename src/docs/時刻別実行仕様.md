# NightlyBatchNotify 時刻別実行仕様

最終更新: 2026-07-15

## 概要

NightlyBatchNotify は、Windows のタスクスケジューラから 15 分ごとに起動されるバッチです。
`main.py` が現在時刻を見て、`config.json` の `batch_schedule` に定義された時刻の
`±batch_window_minutes`(既定14分)以内かどうかを判定し、対象時刻内であれば処理を実行します
(それ以外は何もせず終了)。

タスクスケジューラ自体は登録時刻(0:00)から1日中15分おきに動き続けるため、
`batch_schedule` に時刻を追加・削除するだけで実行タイミングを変更でき、
タスクスケジューラの再登録は不要です(`scheduler\install_scheduled_task.ps1` は
`config.json` の `batch_schedule` を毎回読み直して登録するので、時刻を大きく作り直す
場合のみ再実行すれば十分です)。

## 実行時刻・曜日・実行モジュール(統合スケジュール)

`batch_schedule` の各エントリは **時刻 → 曜日 → 実行モジュール** の3つを1つにまとめた
「スケジュール枠」です。バッチはこの順番で評価します: まず時刻が窓内か、次にその日の
曜日がこの枠の対象か、対象であれば最後にどのモジュールを動かすかを決めます。

```json
{
  "batch_schedule": [
    {
      "time": "07:00",
      "modules": ["stock_nikkei", "stock_watchlist", "stock_sector", "stock_dividend", "market_news", "news_movers", "stock_x_trends", "ai_summary", "report_html", "mail_gmail"]
    },
    {"time": "07:30", "days": "all", "modules": ["post_x_magazine"]},
    { "time": "09:30", "modules": [ /* 07:00と同じレポート系一式 */ ] },
    { "time": "12:15", "modules": [ /* 同上 */ ] },
    {"time": "21:00", "days": "all", "modules": ["post_x_magazine"]},
    { "time": "22:45", "modules": [ /* 同上 */ ] }
  ],
  "batch_window_minutes": 14,
  "batch_weekdays_only": true
}
```

各エントリのフィールド:

- `time`(必須): `"HH:MM"`。省略不可。単純な `"HH:MM"` 文字列だけを書いた場合は、
  `days`・`modules`とも省略したものとして扱われます(例: `"07:00"` は
  `{"time": "07:00"}` と同じ)。
- `days`(省略可): この枠が有効な曜日。省略時は `batch_weekdays_only` の設定に従います
  (`true` なら平日のみ、`false` なら毎日)。指定方法は次項参照。
- `modules`(省略可): この枠で実行するモジュール名の配列。**省略した場合は
  `modules`(有効化フラグ)で `true` になっている全モジュールが対象**になります。

`modules` は基本的に**明示することを推奨**します。省略すると「有効な全モジュール」に
なるため、たとえば `post_x_magazine` を07:00枠に含めるつもりがなくても、有効化フラグが
`true` である限り自動的に対象へ滑り込んでしまいます(モジュール側には可否判定を
持たせない方針なので、これを防ぐ手段は config 側の明示だけです)。現在の設定では
07:00/09:30/12:15/22:45 の4枠すべてに `post_x_magazine` を含まない一覧を明示しているので、
これらの枠では post_x_magazine が呼ばれることすらありません。

現在の設定の内訳:

- `07:00` 朝の総点検(市況・ニュース・Xトレンド検索・メイン配信) — 平日のみ、`post_x_magazine`を除く一式
- `07:30` `post_x_magazine`のみ — **毎日**
- `09:30` 寄り付き後の確認 — 平日のみ、`post_x_magazine`を除く一式
- `12:15` 午前ニュースと午前相場のまとめ — 平日のみ、`post_x_magazine`を除く一式
- `21:00` `post_x_magazine`のみ — **毎日**
- `22:45` 夜の市場データ確認 — 平日のみ、`post_x_magazine`を除く一式

### `days` の指定方法

- 個別の曜日名(3文字・小文字): `"mon"`, `"tue"`, `"wed"`, `"thu"`, `"fri"`, `"sat"`, `"sun"`
  (単一の文字列でも配列でもOK。例: `"days": ["sat", "sun"]`)
- グループ指定(いずれも文字列): `"weekdays"`(月〜金) / `"weekend"`(土日) / `"all"`(毎日)
- 省略時は `batch_weekdays_only` の設定に従います

`post_x_magazine`を除く4枠は市況データが中心なので平日のみ、`post_x_magazine`の2枠は
Xへの投稿が主目的で市況とは無関係なため、`days: "all"` で土日も動くようにしています。

## モジュール内部での時刻自己判定は行わない

以前は `stock_x_trends`(`run_times`)や `post_x_magazine`(`run_times`)が、
`BATCH_SCHEDULE_KEY` 環境変数を見て「今回は実処理をするか」を自分で判定していましたが、
現在はこの仕組みを廃止しています。**実行するかどうかは `batch_schedule` の `modules`
指定だけで決まります**(方針: 実行可否の判断はすべて中央のスケジュール設定に一本化し、
モジュール個別の可否判定は持たせない)。

- `stock_x_trends` は `07:00` の `modules` にしか含まれていないので、09:30/12:15/22:45では
  そもそも呼び出されません(`report_html.py` は `output/stock_x_trends.json` の中身を
  そのまま表示するので、朝の検索結果が終日のメールに引き続き掲載される点は変わりません)。
  Grok API呼び出しは引き続き1日1回だけです。
- `post_x_magazine` は `07:30`/`21:00` の `modules` にしか含まれていないので、他の4枠では
  そもそも呼び出されません。

いずれのモジュールも「呼ばれなければ何もしない」だけで、モジュール側に条件分岐は残って
いません。実行タイミングを変えたい場合は `config.json` の `batch_schedule` だけを編集
すれば十分です。

## モジュール概要

### `stock_nikkei`

日経平均、日経225先物、TOPIX 連動 ETF を取得します。

### `stock_watchlist`

注目銘柄の前日比と、直近10営業日分の日次変化を取得します
(yfinanceから15営業日分取得し、直近11本の終値から10日分の前日比を算出)。

### `stock_dividend`

RYLD / SDIV の配当タイミングを扱います。

### `market_news`

日本株関連のニュースを Google News RSS(Google News / 株探 / Reuters の3系統)から
取得します。

### `news_movers`

ニュース本文から銘柄を照合し、値動き銘柄を作ります。

### `stock_x_trends`

Grok を使って、日本株に関係する X 上の話題銘柄やテーマを拾います。実検索は07:00のみ。

### `ai_summary`

市場データとニュースを AI で要約します。

### `post_x_magazine`

`config.json` の `post_x_magazine.magazines` に登録されたテーマ別マガジンから記事を
ローテーション選択し、Gemini(失敗時は本文ベースのフォールバック)で5本のXスレッドを
生成して投稿します。

- 1本目: フック(数字・体験談ベースの書き出し)+ 導入
- 2本目: 内容の具体点
- 3本目: 実用性・気づき
- 4本目: 読後の持ち帰り + 読者への問いかけ(返信を誘う)
- 5本目: マガジン紹介(他のテーマ別マガジンをプロフィール固定ツイートで案内)+ ハッシュタグ

`POST_X_MAGAZINE_PREVIEW=1` / `POST_X_MAGAZINE_NOTIFY_ONLY=1` を立てるとXには投稿せず、
生成結果のみメール通知できます。成功・失敗いずれの場合も通知メールが送られ、本文には
5投稿すべてが並び、失敗時はどの投稿で失敗したか(`n/5`)と理由が表示されます。

手動で単体実行したい場合は `run_post_x_magazine.bat`(`main.py --force --schedule 07:30`
を実行し、`batch_schedule` の `07:30` エントリの `modules` 指定により post_x_magazine
のみが動く)を使います。

### `report_html`

各モジュールの出力 JSON を HTML にまとめます。

### `mail_gmail`

HTML レポートを Gmail で送信します。`post_x_magazine` の通知メールもこのモジュール内の
`send_html_mail()` を共有して送信します。

### `post_x_note`

note 記事を X に投稿する旧世代の独立ランナーです。`post_x_magazine` に役割を引き継いだため
`modules` には登録されておらず、現在は常時OFFです(`run_post_x_note.bat` から手動実行は可能)。

## 必要な環境変数

```env
GMAIL_ADDRESS=...
GMAIL_APP_PASSWORD=...
MAIL_TO=...
GEMINI_API_KEY=...
GROK_API_KEY=...

# post_x_magazine / post_x_note でXに投稿する場合のみ必要
X_API_KEY=...
X_API_SECRET=...
X_ACCESS_TOKEN=...
X_ACCESS_TOKEN_SECRET=...
```

## 補足

- `main.py` は `--force --schedule HH:MM` で任意の時刻枠を強制実行できます(時刻チェック省略)
- `run_force.bat` はテスト用に07:00/09:30/12:15/22:45または全モジュールを選べる対話式バッチです
  (07:30/21:00枠は未対応。post_x_magazine単体は `run_post_x_magazine.bat` を使ってください)
